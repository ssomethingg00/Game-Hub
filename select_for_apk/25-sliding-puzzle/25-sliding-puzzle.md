## 1. Snapshot

The Sliding Puzzle (also called the 15-Puzzle for its classic 4×4 form) is a single-player
combinatorial puzzle in which numbered tiles fill a grid with one empty space. The player
slides tiles into the blank until all tiles are arranged in ascending numerical order. The
appeal is pure spatial logic: deceptively simple to understand, satisfying to solve, with
depth hidden in the move-count optimisation challenge.

**Quick facts:**
- **Origin:** Noyes Palmer Chapman, ~1874; popularised by Sam Loyd 1880. [Confirmed]
- **Genre:** Sliding tile puzzle / brain teaser.
- **Platforms (modern web clones):** Browser (all devices), occasionally iOS/Android apps. [Confirmed]
- **Age/content rating:** IARC 3+ (no violence, no objectionable content). [Confirmed]
- **Monetization model (this clone):** None — free, ad-free, single HTML file.
- **Comparable games:** 8-Puzzle, 2048, Unblock Me, Sokoban.
- **Session length:** 2–15 minutes per puzzle depending on size and skill. [Estimated]

## 2. Core Loops

- **30-second loop:** Scan the board for a tile adjacent to the blank (same row or column),
  click/tap it to slide it — or an entire row/column segment — into the blank space.
  Increment move counter. Visually check progress toward solved state.

- **Session loop:** Choose grid size (3×3 / 4×4 / 5×5). Press New Game — a solvable scramble
  is generated. Solve the puzzle by sliding tiles. Timer and move counter run live.
  Win overlay fires on solve; best score (moves + time) is saved per size.
  Optionally switch between numbered tiles and a picture tile set.

- **Meta loop:** Chase personal best (fewest moves, fastest time) per grid size. The 3×3 serves
  as a warm-up; 4×4 is the classic challenge; 5×5 is the mastery tier. Returning players are
  motivated by improving their best scores stored in localStorage.

## 3. Mechanics, Controls & Game States

### Mechanics

| Mechanic | Detail |
|---|---|
| Tile slide | Click/tap a tile in the same row or column as the blank; it slides (with all tiles between it and the blank) into the blank position. [Confirmed] |
| Keyboard control | Arrow keys move the blank: ← → ↑ ↓ shift the blank one step, pulling the adjacent tile into it. [Estimated] |
| Solvable scramble only | Scrambles are generated by applying N random legal moves from the solved state (guarantees solvability). Alternatively verified via inversion-parity check. [Confirmed] |
| Win detection | After every move, compare the flat tile array to [1,2,3…N²-1, 0]; if equal, puzzle is solved. [Confirmed] |
| Size selector | 3×3 (8-puzzle), 4×4 (15-puzzle), 5×5 (24-puzzle). Choosing rebuilds the board from scratch. [Estimated] |
| Tile type toggle | "Numbers" mode: tiles show their number. "Picture" mode: tiles show a fragment of a CSS-drawn geometric picture; blank stays blank. [Estimated] |
| Move counter | Integer, increments +1 per move (one tap/click = one move, regardless of how many tiles slide). [Confirmed] |
| Timer | MM:SS elapsed since first move, stops on solve. Resets on New Game. [Estimated] |
| Best tracking | Per-size localStorage record: fewest moves + fastest time. [Estimated] |

### Core Verbs
Click/tap, slide, plan.

### Game Modes

| Mode | Description |
|---|---|
| Free play (standard) | Choose size, solve at own pace, no lives, no fail state. |
| New Game (restart) | Instantly regenerates a new solvable scramble for current size. |

### Input Scheme
- **Mouse:** Left-click a tile in the blank's row or column to slide it.
- **Touch:** Tap (mobile). Same row/column rule applies.
- **Keyboard:** Arrow keys (move blank direction); tiles slide toward blank accordingly.
- **Orientation:** Portrait-primary on mobile; landscape fine on desktop. Board is square, scales to fit.

### Win / Lose / Fail Conditions
- **Win:** Tiles in order [1…N²-1] with blank at bottom-right. Win overlay shows time + moves + best. Replay button available.
- **No lose/fail state:** Puzzle has unlimited moves and time; there is no penalty.

### AI / Opponent

### Difficulty

### Feedback Systems

## 5. Economy & RNG

### Currencies

No in-game economy. There are no currencies, coins, or premium items. [Confirmed]

### RNG

| RNG element | Method | Notes |
|---|---|---|
| Scramble generation | Random-legal-moves walk from solved state | Apply 200+ random legal moves; result is always solvable by construction. [Confirmed] |
| Scramble minimum complexity | Min-move threshold enforced | Reject scrambles solvable in <20 moves (re-scramble); ensures puzzle is non-trivial. [Estimated] |
| No other RNG | — | No drops, no gacha, no loot. [Confirmed] |

### Solvability Guarantee (Technical)

**Odd-width grids (3×3, 5×5):** Solvable iff inversion count is even.
**Even-width grids (4×4):** Solvable iff (inversions + blank_row_from_bottom) is even.

Formula: `isSolvable = (width % 2 === 1) ? (inversions % 2 === 0) : ((inversions + blankRowFromBottom) % 2 === 0)` [Confirmed]

The random-legal-moves approach avoids needing to check this at all — every generated state is reachable from solved. [Confirmed]

## 15. Pacing & Difficulty

### Early Game (3×3, first plays)
- Puzzle solved in 10–60 moves for new players. [Estimated]
- "Aha" moment: realising you can slide a whole segment by clicking a distant tile.
- Most players solve 3×3 within 2–3 attempts. [Estimated]

### Mid Game (4×4, practiced player)
- 80–150 moves to solve; systematic row-by-row technique required. [Estimated]
- Key milestone: understanding the 3-cycle rotation technique for the bottom rows.
- Churn point: players who don't discover technique may get frustrated and quit; hint text helps.

### Late / Mastery (5×5 + optimisation)
- 200–400 moves; demands pre-planning. [Estimated]
- Re-play motivation: chasing sub-100-move 4×4 solutions.
- Stall point: none for casual players (no timer pressure, no fail state); hardcore players self-impose move-count challenges.

### Difficulty Curve
```
3×3 [~30s]  →  4×4 [~5min]  →  5×5 [~15min]
Easy             Medium           Hard
```
Progression is purely player-driven; no adaptive difficulty needed.

### Phased Roadmap

**Phase 1 — Playable core (MVP)**
1. HTML skeleton: viewport meta, title, CSS reset.
2. Board renderer: generate `N×N` grid of `<div>` tiles from state array.
3. State: flat array `[1,2,...,N²-1,0]`; track blank index.
4. Scramble: apply 500 random legal moves from solved state.
5. Click handler: check if clicked tile is in same row/column as blank; slide all tiles between.
6. Move counter: increment on each valid move.
7. Win detection: compare state to goal after each move; show basic alert.

**Phase 2 — Polish + completeness**
8. CSS transitions for smooth tile sliding.
9. Timer (start on first move, stop on win).
10. Win overlay (modal with moves, time, replay button).
11. Size selector buttons (3/4/5) — rebuild board on click.
12. localStorage best tracking.
13. Keyboard arrow key support.
14. Responsive layout: board scales to viewport.

**Phase 3 — Juice + extras**
15. Hover highlight for eligible tiles.
16. Picture tile mode (CSS background slicing).
17. Web Audio SFX (slide click, win fanfare).
18. Mute toggle + persist.
19. Confetti win animation.
20. Hint text with auto-hide.
21. Accessibility: aria-labels, prefers-reduced-motion.

### Recommended Tech Stack
- Pure HTML5 / CSS3 / ES6+ JavaScript.
- No framework, no library, no bundler.
- Web Audio API for sound synthesis.
- CSS Grid for board layout.
- CSS custom properties for theming.
- `localStorage` for persistence.

### Required Asset List
- No external assets required.
- Picture tiles: CSS gradient/geometric art generated inline.
- Fonts: system stack (`-apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif`).
- Audio: synthesised via Web Audio API.

### Hardest Parts / Risks

| Risk | Mitigation |
|---|---|
| Sliding a whole segment correctly | Slide ALL tiles between clicked tile and blank as a unit; re-index blank position after |
| Solvable scramble guarantee | Use legal-moves random walk (not Fisher-Yates on array) — trivially solvable |
| Smooth tile animation with DOM | Use CSS `transition` on `transform: translate()` or swap grid positions with `order`; transitions need a reflow trick |
| 5×5 performance | 24 divs is trivial; no canvas needed |
| Touch vs click deduplication | Use `pointerdown` or separate touch/click handlers to avoid double-firing on mobile |
| Win detection false negative | Compare every element including blank (index N²-1 = 0) to goal array |

## 17. Open Questions

1. **Optimal move display:** Should the HUD also show the theoretical minimum moves (BFS
   solver) to give a "efficiency %" stat? Adds a background solver — out of scope for Phase 1
   but would be a compelling feature. [Needs decision]

2. **Picture content:** The CSS-drawn art is abstract/geometric to avoid any IP concern.
   A future version might allow the user to paste an image URL; requires CORS handling. [Needs decision]

3. **5×5 scramble depth:** 500 random legal moves may occasionally leave the 5×5 in a
   near-solved state. Increasing to 1000 moves reduces this; minimum-move threshold check
   recommended. [Estimated; needs tuning]
